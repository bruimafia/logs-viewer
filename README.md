# Просмотрщик логов

Веб-приложение для просмотра и анализа лог-файлов в формате JSON Lines с поддержкой удалённых серверов по SSH/SFTP и локальных файлов через Node.js. Используется для микросервисов Перекоп-И 2.

## Основные возможности

- **Источники данных**
  - Локальные файлы — drag-and-drop / input выбор `.log` / `.json`.
  - Удалённые SSH/SFTP-серверы.
  - «Локальные» серверы (`type: "local"`) — файлы на машине, где запущен Node.js, читаются напрямую через `fs`, без SSH.
  - Поддержка **glob-паттернов** (`*`, `?`) в путях; раскрытие в список конкретных файлов выполняется на сервере (SSH `ls -1d` или Node `fs`).
- **Три режима загрузки удалённых файлов** (в модалке по умолчанию выбран **Live**)
  - **Live** — стриминг новых строк в реальном времени через `tail -F` (переживает ротацию).
  - **Хвост** — последние N строк через `tail -n` с пагинацией «↑ Загрузить ещё ↑».
  - **Диапазон дат** — потоковое чтение файла через SFTP с фильтрацией по полю `time` каждой строки.
  - Опциональный **серверный grep** для «Хвоста» и «Диапазона» — фильтрация выполняется на сервере до сетевой передачи.
  - Опциональный **серверный фильтр по уровню** для «Диапазона» — `grep -E '"level":"(ERROR|WARN)"'` экономит трафик на больших файлах.
- **Управление серверами и файлами**
  - Все настройки доступны прямо в UI: вкладка **«⚙ Серверы»** в модалке удалённых файлов — добавление, редактирование, удаление серверов и файлов, glob-превью, проверка SSH-соединения.
  - Альтернатива — править `remote-config.json` руками.
- **Работа со списком логов**
  - Объединение записей из нескольких файлов в единый хронологический поток.
  - **Виртуализация** — в DOM остаются только видимые строки плюс буфер (~30 сверху и снизу); десятки тысяч записей прокручиваются без подвисаний.
  - Поиск по `msg` и по сериализованной записи целиком с подсветкой совпадений (включая раскрашенные поля JSON в блоке «Дополнительные поля»).
  - Фильтр по уровню: ERROR / WARN / INFO / DEBUG (чекбоксы в тулбаре или клик по бейджу уровня в строке).
  - Фильтр по сервисам через цветные чипы (клик скрывает/показывает) или клик по имени сервиса в строке; кнопка с иконкой глаза — скрыть или показать все сервисы разом.
  - Фильтр по временно́му диапазону с пресетами (5 м / 15 м / 30 м / 1 ч /
    3 ч / 6 ч / 12 ч / 24 ч / 7 дн / Сегодня / Вчера).
  - Сортировка по времени, сервису, уровню или трассе.
  - **Группировка и трассировка по `traceId` / `requestId`** — клик по бейджу собирает только нужную трассу; сортировка «По трассе» соединяет записи одной трассы подряд. В баннере активного фильтра — **иконка [Jaeger](https://www.jaegertracing.io/)**: открытие этой трассы в Jaeger UI в новой вкладке (URL настраивается в коде).
  - **Цветовая подсветка сервисов** — стабильный цвет, детерминированно вычисляемый из имени сервиса; чипы и колонка «сервис» подсвечиваются одинаково.
- **Live-режим**
  - Баннер подключения над списком при старте: статус каждого файла (подключение, успех, ошибка) и анимированный индикатор прогресса; скрывается автоматически, когда все потоки готовы.
  - Несколько одновременных потоков, мультиплексированных в одно SSE-соединение на сервер (обходит лимит браузера в 6 HTTP/1.1-сокетов).
  - SSH-соединения объединены в пул: до 8 каналов на одно соединение, автоматическое закрытие при простое в 5 секунд.
  - Пауза без разрыва соединения: новые строки копятся в буфере, применяются на «Возобновить».
  - Скользящее окно — общий буфер ограничен **50 000 записями**.
  - Авто-скролл к новым записям; отключается при ручной прокрутке.
  - **Алерты на ERROR** — браузерные desktop-уведомления (Notification API) и/или короткий звуковой сигнал (Web Audio API). Дросселируются, не срабатывают на первый исторический батч и на паузе.
- **Спарклайн активности** — гистограмма количества записей по уровням за выбранное окно (5 мин — 24 ч); то же окно задаёт временные границы списка логов. Клик по бину сужает диапазон «С / До» до границ бина.
- **Часовой пояс отображения** — селектор в шапке: «Локальное», UTC, фиксированные смещения. Влияет на колонку времени и подсказку.
- **Компактный режим** — уменьшает высоту строк и шрифт до 11 px.
- **Тёмная / светлая тема** — переключается одной кнопкой, выбор сохраняется в `localStorage`, применяется до первого paint (без вспышки).
- **Toast-уведомления** — неблокирующая замена `alert()` / `confirm()`, окрашены по уровню (error/warn/info/success), скрываются автоматически.
- **Доступность** — skip-link, ARIA-роли для тулбара, live-region для статистики, поддержка `prefers-reduced-motion`.

## Требования

- **Node.js 18+** (`node:test` требует 18; для запуска без тестов достаточно 16+).
- **Удалённые серверы:** Linux/Unix с `tail` (`-n`, `-F`) и `grep` (`-E`, `-F`, `-i`, `-a`). Стандартный coreutils подходит. Поддержка `grep` обязательна, если используются серверный поиск или фильтр по уровню.
- **SSH-доступ** должен разрешать выполнение exec-команд, не только SFTP.
- **Права доступа:** пользователь, под которым выполняется подключение или запущен Node.js, должен иметь доступ на чтение файлов логов.
- **«Локальные» серверы** работают на любой ОС, где запущен Node.js (Windows / Linux / macOS).
- **Современный браузер:** ES-модули, `fetch` с `ReadableStream` (для SSE), `color-mix()` для подсветки. Chrome / Edge / Firefox / Safari последних 5 лет — без сборщика.

## Установка и запуск

```bash
npm install
npm start          # сервер на http://localhost:5000
npm run dev        # то же самое
```

После запуска откройте <http://localhost:5000>.

## Конфигурация серверов

### Способ 1: через UI

В модалке «Удалённые файлы» переключитесь на вкладку **⚙ Серверы**:

- **+ Добавить сервер** — выбор типа (SSH / Локальный), поля хост/порт/пользователь/пароль, проверка соединения.
- **+ Добавить файл** — название и путь (поддерживаются glob-паттерны, кнопка «Просмотр» сразу показывает список совпадающих файлов).
- Карандаш / крестик — редактирование и удаление.

Примеры glob-паттернов: для локального сервера можно использовать `C:\perekop-2-server\microservices\*\*.log`, для SSH-сервера — `/var/www/services/*/*.log`.

Изменения немедленно сохраняются в `remote-config.json` через `POST /api/config`. Пароль при отображении маскируется как `••••••••`; если оставить это значение при сохранении, прежний пароль не меняется.

### Способ 2: ручная правка `remote-config.json`

Файл лежит в корне проекта:

```json
{
  "servers": [
    {
      "id": "stage",
      "name": "Stage Server",
      "host": "192.168.185.111",
      "port": 22,
      "username": "user",
      "password": "P@ssw0rd",
      "files": [
        {
          "id": "order",
          "name": "Order Service Log",
          "remotePath": "/var/www/services/order-service/order-service.log"
        },
        {
          "id": "all-services",
          "name": "Все сервисы",
          "remotePath": "/var/www/services/*/*.log"
        }
      ]
    }
  ]
}
```

После правки руками можно либо перезапустить процесс, либо вызвать `GET /api/reload-config` — конфиг перечитывается с диска, неиспользуемые SSH-пулы закрываются.

#### Поля сервера

| Поле       | Описание                                                          |
| ---------- | ----------------------------------------------------------------- |
| `id`       | Уникальный идентификатор сервера                                  |
| `name`     | Отображаемое имя                                                  |
| `type`     | `"local"` для локального сервера; для SSH — отсутствует           |
| `host`     | IP-адрес или hostname (только для SSH)                            |
| `port`     | SSH-порт, обычно `22`                                             |
| `username` | Имя пользователя SSH                                              |
| `password` | Пароль SSH. `••••••••` интерпретируется как «не менять»           |
| `files`    | Массив описаний файлов                                            |

#### Поля файла

| Поле         | Описание                                                       |
| ------------ | -------------------------------------------------------------- |
| `id`         | Уникальный идентификатор файла                                 |
| `name`       | Отображаемое имя                                               |
| `remotePath` | Полный путь, либо glob-паттерн (`*`, `?`)                      |

### Локальный сервер (`type: "local"`)

Для просмотра файлов на той же машине, где запущен Node.js, добавьте сервер с `type: "local"`:

```json
{
  "id": "localhost",
  "name": "Локальный сервер",
  "type": "local",
  "files": [
    {
      "id": "all-logs",
      "name": "Все логи Перекоп",
      "remotePath": "C:\\perekop-2-server\\microservices\\*\\*.log"
    }
  ]
}
```

Для локального сервера хост/пользователь/пароль игнорируются. Все три режима (Хвост, Диапазон, Live) реализованы без SSH:

- Хвост — читается через `fs.createReadStream`, последние N строк выбираются построчным буфером.
- Диапазон — потоковое чтение `fs.createReadStream` с фильтрацией по `time` каждой строки.
- Live — polling через `fs.statSync` каждые 250 мс с дочитыванием новых байт.

### Glob-паттерны в `remotePath`

Поле `remotePath` поддерживает подстановки `*` и `?`:

| Паттерн                          | Смысл                                                |
| -------------------------------- | ---------------------------------------------------- |
| `/var/log/app/*.log`             | Все `.log` в директории                              |
| `/var/log/services/*/*.log`      | Все `.log` во вложенных директориях `services/*/`    |
| `/var/log/app/service-*.log`     | Файлы вида `service-api.log` и т. п.                 |
| `C:\logs\*\*.log`                | На Windows: все логи в поддиректориях `C:\logs\`     |

Поведение:

- Glob раскрывается при открытии модалки «Удалённые файлы» через `POST /api/expand-glob` (SSH-серверу — `bash -c 'ls -1d <pattern>'`, локальному — Node `fs.readdirSync`).
- Каждый найденный файл становится отдельной строкой со своим чекбоксом и **автоматически сгенерированным именем**: `/var/log/order-service/app.log` → «Order Service Log». Generic-имена (`app`, `service`, `main`, `log`? …) заменяются на имя родительской директории. Имена в стиле дат (`2024-01-15.log`) — тоже.
- При загрузке через API конкретный путь отдаётся как `overridePath`, не меняя сам конфиг.

## Использование

### Локальные файлы

1. **«Открыть файл(ы)»** → выберите один или несколько `.log` / `.json`.
2. Чекбокс **«Добавить к открытым»** управляет режимом: добавить к существующим или заменить.

### Удалённые файлы

1. **«Удалённые файлы»** → откроется модальное окно.
2. Сверху выберите **режим загрузки** (см. ниже).
3. Дождитесь автоматической проверки соединений (зелёный «Подключено» рядом с каждым сервером).
4. Клик по заголовку сервера раскрывает список файлов (по умолчанию свёрнут). Отметьте чекбоксы нужных файлов; кнопка «Выбрать все» — для всего сервера.
5. Нажмите кнопку загрузки внизу; её текст меняется по режиму («Запустить Live» / «Загрузить хвост» / «Загрузить по диапазону»).

### Режим «Live» (по умолчанию)

Стриминг через `tail -n N -F` (`-F` переоткрывает файл после ротации).

- **«Начальные строки»** (по умолчанию 30) — сколько последних строк показать в момент старта. `0` — только новые события.
- **Баннер подключения** — над списком появляется панель со статусом каждого выбранного файла (спиннер «Подключение…», галочка при успехе, сообщение об ошибке). После готовности всех потоков панель исчезает сама.
- **Мультиплексирование:** все файлы одного сервера передаются через одно SSE-соединение (`/api/tail-follow-multi`); это обходит браузерный лимит в 6 одновременных HTTP/1.1-сокетов и держит общее число открытых SSH-сессий минимальным.
- В шапке появляется красный пульсирующий индикатор **`● LIVE [N] [Пауза] [Стоп]`**:
  - Клик по индикатору — список активных потоков с возможностью остановить каждый по отдельности.
  - **Стоп** — остановить все потоки и очистить логи (`Очистить все`).
  - **Пауза** — приостановить отображение, не разрывая соединение: серверные процессы продолжают работать, новые строки копятся в state.livePausedBuffer`. На кнопке появляется счётчик (`▶ Возобновить (+42)`). При возобновлении буфер применяется одним пакетом; при «Стоп» или «Очистить все» — отбрасывается.
- **Алерты на ERROR** — два независимых чекбокса 🔔 / 🔊 в шапке (см. раздел «Уведомления»).
- Модальное окно можно закрывать — потоки продолжат работать в фоне.
- Можно одновременно стримить **несколько файлов** с разных серверов.
- При появлении новые записи кратко подсвечиваются зелёной вспышкой.
- При активных потоках общий буфер ограничен **50 000 записями** — самые старые отбрасываются автоматически (скользящее окно).
- Авто-прокрутка к новым записям работает, **пока вы не прокрутили список вручную**; чтобы включить обратно — прокрутите к краю с новейшими записями.

### Режим «Хвост»

Последние N строк через `tail -n` по SSH (или Node `fs` для локальных серверов). Не зависит от размера файла.

- Параметр **«Строк»** (по умолчанию 100).
- Во время загрузки показывается прогресс-бар со статусом («Загрузка хвоста…», «Получено строк: N»).
- После загрузки появляется кнопка **«↑ Загрузить ещё (более старые) ↑»**
  — подгружает следующую страницу той же длины **для всех** ранее загруженных в режиме «Хвост» файлов (`tail -n (N+offset) | head -n N`).
- Если новых строк нет — достигнуто начало файла (toast «Конец файла»).

### Режим «Диапазон дат»

Полное чтение файла через SFTP с построчным фильтром по `time`.

- **«От»** и **«До»** — границы периода (`datetime-local`, точность до секунды). Оба пустые = весь файл.
- **Фильтр по уровню (на сервере)** — чекбоксы ERROR/WARN/INFO/DEBUG. Когда хотя бы один уровень снят, вместо SFTP-чтения запускается `grep -E '"level":"(ERROR|WARN)"'` через SSH exec: по сети передаются только нужные строки. Радикально ускоряет работу с большими файлами.
- Во время загрузки — прогресс-бар: процент по байтам при SFTP-чтении или индетерминантный индикатор при серверном grep.

### Серверный поиск (опционально)

В модалке между селектором режима и его настройками — поле **«Содержит»**.

- Чекбокс **`.* regex`** — POSIX ERE (`grep -E`); по умолчанию литеральный поиск (`grep -F`) — `.`, `*`, `+`, `[`, `]` интерпретируются буквально.
- Чекбокс **`Aa`** (включён по умолчанию) — `grep -i`.
- Используется для режимов «Хвост» и «Диапазон»; в «Live» поле скрывается.
- В режиме «Хвост» меняется семантика: вы получаете последние N **совпавших** строк, а не последние N строк среди которых может быть 0 совпадений. Кнопка «↑ Загрузить ещё» сохраняет тот же фильтр.
- В режиме «Диапазон» прогресс становится индетерминантным (общий размер источника после grep неизвестен).
- Ошибки grep (некорректный regex, нет прав, и т. п.) показываются как toast.

### Совместное использование режимов

- **«Добавить к открытым»** работает во всех трёх режимах: можно загрузить хвост за вчерашний день и поверх запустить live.
- При снятой галочке новая загрузка очищает всё, **в том числе останавливает все live-потоки** и **снимает активный фильтр по трассе**.
- **«Очистить все»** в шапке делает то же самое явно.

## Фильтрация и сортировка

Панель фильтров под шапкой работает на стороне клиента поверх всех
загруженных записей:

- **Поиск** — подстрочный, регистронезависимый, по `msg` и по сериализованной записи целиком. Совпадения подсвечиваются жёлтым `<mark>` и в столбце сообщения, и в блоке «Дополнительные поля» (поверх синтаксической раскраски JSON).
- **Уровни** — чекбоксы ERROR / WARN / INFO / DEBUG. Клик по цветному бейджу уровня в строке включает соло-фильтр «только этот уровень» (чекбоксы синхронизируются); повторный клик по тому же бейджу снимает фильтр и включает все уровни.
- **С / До** — `datetime-local` с точностью до секунды; пересекаются с окном спарклайна «Активность» (см. ниже). Без ручного диапазона список ограничен только окном спарклайна.
- **Быстрый диапазон** — выпадающий список:
  - «Последние»: 5 минут / 15 минут / 30 минут / 1 час / 3 часа / 6 часов / 12 часов / 24 часа / 7 дней.
  - «Календарь»: Сегодня (от локальной полуночи до текущего момента), Вчера (полные сутки от полуночи до полуночи).
  - Кнопка **×** очищает диапазон. Ручное редактирование полей «С / До» сбрасывает выбранный пресет.
- **Сортировка** — по времени (старые↑ / новые↑), по сервису, по уровню, по трассе (см. ниже).
- **Сервисы** — чипы; клик по чипу скрывает/показывает логи сервиса (зачёркнутый чип = скрытый). Клик по имени сервиса в строке — соло-фильтр «только этот сервис»; повторный клик снимает фильтр и включает все сервисы. Кнопка с иконкой глаза слева от чипов переключает все сервисы разом («Скрыть все» / «Отобразить все»). У каждого сервиса свой **стабильный цвет** (см. ниже).

### Группировка и трассировка по `traceId` / `requestId`

Если запись содержит идентификатор трассы, в колонке **trace** появляется небольшой **цветной бейдж** с укороченным ID. Цвет стабильно генерируется из самого ID, поэтому записи одной трассы визуально одного цвета.

- **Клик по бейджу** — фильтр «показать только эту трассу». Между тулбаром и списком появляется баннер `Фильтр по трассе: <id>` с **иконкой Jaeger** справа от полного ID и кнопкой **×**. Иконка открывает трассу в **Jaeger UI** в новой вкладке (см. ниже про URL). Кнопка **×** или повторный клик по тому же бейджу — снять фильтр; прокрутка возвращается к той же записи, которая была у верхнего края списка до включения фильтра.
- **Сортировка «По трассе»** — записи одного `traceId` идут подряд, внутри группы — по времени. Между группами — по времени самой ранней записи. Записи без трассы становятся одиночными «группами» по своему времени, а не сливаются в одну смешанную пачку.

Распознаваемые поля идентификатора (в порядке приоритета — побеждает первое непустое значение):

```
traceId  →  trace_id  →  requestId  →  request_id  →  correlationId  →  correlation_id
```

Поддерживаются строковые и числовые значения (числа приводятся к строке); объекты, массивы, булевы и `NaN` пропускаются. Если ни одно поле не найдено, бейдж не отображается.

Полный исходный `traceId` всегда виден в раскрывающемся блоке **«Дополнительные поля»** — оттуда его удобно скопировать вручную (например, для Tempo или другого UI). Для **Jaeger** быстрее воспользоваться иконкой в баннере фильтра по трассе.

**Шаблон URL Jaeger** задаётся константой `JAEGER_URL_TEMPLATE` в `public/state.js`. По умолчанию: `http://{ip}:16686/trace/{traceId}` — подставляются текущий идентификатор трассы и **IP или hostname сервера**, с которого пришли логи (берётся из первой записи с этим `traceId` среди загруженных данных; для локальных файлов без удалённого сервера используется hostname страницы в браузере). При необходимости замените шаблон на свой (другой порт, HTTPS, reverse-proxy и т. д.).

### Подсветка сервисов цветом

Каждому сервису назначается **стабильный цвет** `hsl(hue, 55%, 50%)`, где `hue` детерминированно вычисляется из имени сервиса через 32-битную хэш-функцию. Один и тот же сервис всегда выглядит одинаково в разных сессиях.

Цвет применяется одновременно в двух местах:

- **Чипы сервисов** в панели фильтров — лёгкий цветной тинт фона, граница, цвет текста.
- **Колонка «сервис»** в каждой строке лога — те же тинт и граница.

Параметры HSL специально отличаются от бейджей трасс (`hsl(hue, 60%, 55%)`), чтобы визуально сервисы и трассы не путались.

Скрытый чип сервиса теряет цветовой акцент и становится нейтрально-серым, иначе он бы продолжал привлекать внимание, хотя его данных в списке нет.

Для старых браузеров без `color-mix()` (старше ~2023 г.) работает фолбэк: чипы и колонка сервиса отрисовываются нейтральным цветом.

### Строка лога

Каждая запись — grid из колонок: **время** | **уровень** | **сервис** | **trace** | **доп. поля** | **сообщение**. Бейджи уровня, сервиса и trace кликабельны (см. фильтры выше). Блок **«Дополнительные поля»** открывается иконкой с счётчиком полей и раскрывается отдельной строкой под записью. На узких экранах (< 900 px) колонки перестраиваются в несколько строк.

## Спарклайн активности

Узкая полоса под тулбаром — гистограмма количества записей по уровням ERROR/WARN/INFO/DEBUG за выбранное окно. Все логи делятся на 60 бинов; бары стэкуются «снизу-вверх»: DEBUG → INFO → WARN → ERROR.

- Селектор **«Окно»**: 5 мин / 15 мин / 30 мин / 1 ч / 6 ч / 24 ч (по умолчанию — 30 минут, сохраняется в `localStorage`). Смена окна сразу пересчитывает видимый список логов.
- В live-режиме окно «едет» вправо по `Date.now()` с автотиком раз в 5 с; без live правый край — время самой свежей записи.
- **Гистограмма** строится по всем загруженным записям внутри окна и **не** зависит от поиска, уровней, сервисов и фильтра по трассе — так видно полную картину активности.
- **Список логов** ограничен тем же временным окном; поля «С / До» могут только сузить его (пересечение диапазонов), но не расширить за пределы окна.
- Цвета берутся из CSS-переменных `--error/--warn/--info/--debug`, поэтому при смене темы спарклайн перерисовывается через `MutationObserver` на `<html data-theme>`.
- Сводка в правой части шапки — счётчики по уровням и общая сумма.
- **Tooltip при наведении** — точные временны́е границы бина, по уровням и сумма.
- **Клик по бину** — выставляет «С / До» на границы бина (с небольшим запасом), сужая видимый диапазон внутри окна.
- Кнопка **«Свернуть»** прячет канвас, оставляя только сводку (состояние сохраняется).

## Часовой пояс отображения

Селектор в шапке (значок 🕓 рядом со счётчиком открытых файлов):

- **Локальное** — браузерный пояс (поведение по умолчанию). Ярлык динамически дополняется текущим UTC-смещением, чтобы было видно, что именно за ним стоит.
- **UTC** — всегда UTC+00:00.
- Фиксированные смещения — UTC+01:00 … UTC+12:00, UTC−01:00 … UTC−08:00, плюс редкие (UTC+04:30, UTC+05:30 и т. п.).

Изменение применяется и к колонке времени, и к подсказке с полной временно́й меткой. Выбор сохраняется в `localStorage['tz-mode']`.

## Компактный режим

Чекбокс **«Компактно»** в шапке уменьшает паддинги списка, шрифт до 11 px и сжимает столбец сервиса (длинные имена обрезаются многоточием, полное имя — в `title`). Состояние сохраняется в `localStorage['compact-mode']` и применяется до первого paint inline-скриптом в `<head>`, поэтому геометрия не «прыгает» после первой перерисовки.

## Переключение темы

Кнопка с иконкой солнца/луны в правой части шапки переключает интерфейс между **тёмной** (по умолчанию) и **светлой** темой. Выбор сохраняется в `localStorage['theme']` и применяется до первого paint (через синхронный inline-скрипт в `<head>` — `<script type="module">` имеет defer-семантику и не успеет до отрисовки `<body>`).

## Уведомления

### Toast

Все нефатальные ошибки, успехи и подтверждения показываются как **toast-уведомления** в правом нижнем углу — поверх контента, но не блокируют интерфейс.

- **error** (красный) — неудачная загрузка, разрыв live, ошибка SSH.
- **warn** (жёлтый) — подтверждения деструктивных действий (`toastConfirm`).
- **info** (синий) — например, «Конец файла» в режиме «Загрузить ещё».
- **success** (зелёный) — например, «Сервер обновлён».

Длительность зависит от уровня (3.5 — 6 с); при наведении курсора таймер ставится на паузу. Кнопка «×» закрывает уведомление вручную. Toast-подтверждения с кнопками «Подтвердить» / «Отмена» ждут явного ответа (`toastConfirm()`): `Esc` или крестик равны «Отмена».

### Алерты на ERROR в live-режиме

В шапке — две независимые галочки:

- 🔔 **Уведомления** — браузерные desktop-уведомления через Notification API. При первом включении браузер запросит разрешение. Показываются **только когда вкладка скрыта** (`document.hidden`). Клик по уведомлению фокусирует вкладку.
- 🔊 **Звук** — короткий двухтональный «бип» (880 Hz → 660 Hz, ~200 мс) через Web Audio API. Без внешних аудио-файлов. При включении сразу проигрывается тестовый сигнал.

Поведение:

- Срабатывают **только при активных live-потоках и не на паузе**.
- **Первый исторический батч** строк (`initialLines`) игнорируется — это уже произошедшие события.
- Дросселирование: уведомления — не чаще раза в 5 с, звук — не чаще раза в 2 с.
- Учитываются записи с уровнем `ERROR`, `ERR`, `FATAL` или `CRITICAL`.
- Настройки сохраняются в `localStorage` и восстанавливаются при перезагрузке.
- Если уведомления заблокированы на уровне браузера, галочка автоматически блокируется с поясняющим `title`.

## Подсказка с полной временно́й меткой

Колонка времени в каждой строке лога показывает компактное локализованное время `dd.mm.yyyy, hh:mm:ss`. При наведении курсора появляется **нативный tooltip** с четырьмя строками:

```
понедельник, 15 января 2024 г.
10:30:45.123 (UTC+03:00)
ISO: 2024-01-15T07:30:45.123Z
5 минут назад
```

- **Полная дата** — на случай, когда время лежит на границе суток.
- **Локальное время с миллисекундами + UTC-смещение** — миллисекунды для быстрых event'ов, смещение — когда логи приехали из другой таймзоны. Если в селекторе выбран не «Локальное», и время, и смещение пересчитываются в выбранный пояс.
- **ISO 8601** — всегда в UTC; удобно скопировать в фильтр другой системы.
- **Относительное время** — «5 минут назад», «2 часа назад», «через 30 секунд» (для записей из будущего, например с серверов с расходящимися часами). Граница «только что» — ±5 секунд. Поддерживается русское склонение one/few/many.

Записи без поля `time` показываются как `—` и подсказку не получают.

## Формат логов

Приложение ожидает **JSON Lines** — один JSON-объект на строку:

```json
{"time":"2024-01-15T10:30:00.000Z","level":"INFO","msg":"Application started","service":"app"}
{"time":"2024-01-15T10:30:01.000Z","level":"ERROR","msg":"Connection failed","service":"db","error":"timeout","traceId":"5a8e1f4b-9c3d-4f6a-b2e8-1d7f0a3c9b5e"}
```

### Распознаваемые поля

| Поле             | Назначение                                                                 |
| ---------------- | -------------------------------------------------------------------------- |
| `time`           | Временна́я метка в ISO 8601 (используется для сортировки и фильтра)         |
| `level`          | Уровень: `ERROR`, `WARN`, `INFO`, `DEBUG`                                  |
| `msg`            | Основное сообщение                                                         |
| `service`        | Имя сервиса (опционально; иначе — имя файла без расширения)                |
| Поля трассировки | `traceId`, `trace_id`, `requestId`, `request_id`, `correlationId`, `correlation_id` |

Все остальные поля показываются в раскрывающемся блоке **«Дополнительные поля»** с синтаксической раскраской JSON (ключи / строки / числа / `true`/`false`/ `null` — в разных цветах).

Строки, которые не парсятся как валидный JSON, **молча игнорируются**.

### Параметры SSE-эндпоинтов

Все эндпоинты в случае `overridePath` подменяют им `file.remotePath` — это используется при загрузке glob-расширенного файла, чтобы передать конкретный путь, не меняя конфиг.

**`/api/stream-file`** — `{ serverId, fileId, dateFrom?, dateTo?, logLevels?, grepPattern?, grepRegex?, grepCaseInsensitive?, overridePath? }`. Ветвление по приоритету:

- **`grepPattern`** задан → `grep [-E|-F] [-i] -a -e <pattern> <path>` через SSH exec. По сети передаются только совпадения; фильтр по дате применяется построчно. Прогресс индетерминантный (`totalBytes: 0`).
- иначе **`logLevels`** задан → `grep -E '"level":"(ERROR|WARN)"' <path>`. То же индетерминантное поведение прогресса.
- иначе **SFTP-поток** через `ssh2-sftp-client.createReadStream` с построчным фильтром по дате и нормальным прогрессом по байтам.

Локальный сервер реализует все три ветки на `fs.createReadStream`, без SSH/SFTP.

События: `start` (с флагами `grepFilter`/`levelFilter`/`local`), `progress`, `lines`, `complete`, `error`.

**`/api/tail-file`** — `{ serverId, fileId, lines, offsetLines?, grepPattern?, grepRegex?, grepCaseInsensitive?, overridePath? }`. На сервере выполняется:

- без grep: `tail -n (lines+offsetLines) <path> | head -n lines` (на `offsetLines = 0` — просто `tail -n lines`).
- с grep: `grep ... <path> | tail -n ... | head -n ...` — «последние N **совпавших** строк». В этом режиме непустой stderr трактуется как ошибка независимо от exit-кода (иначе ошибка grep потерялась бы за exit-кодом tail).

События: `start` (с `grepFilter`/`local`), `lines`, `complete`, `error`.

**`/api/tail-follow-multi`** — `{ serverId, files: [{ fileId, fileKey?, initialLines?, overridePath? }, ...] }`. Открывает **одно** SSE-соединение, в котором мультиплексируются стримы нескольких файлов одного сервера. Каждые 25 секунд шлётся комментарий `: keepalive` для обхода прокси-таймаутов.

События (поле `fileId` в каждом — это `fileKey`, если он передан):

- `start` — `{ groupId }`
- `file-start` — `{ fileId, fileName, filePath, serverName, mode: 'live', initialLines, local? }`
- `file-lines` — `{ fileId, lines: [...] }`
- `file-info` — `{ fileId, message }` (например, ротация)
- `file-error` — `{ fileId, message }`
- `file-end` — `{ fileId, exitCode }`
- `group-end` — `{}`

Остановка отдельного канала: `POST /api/tail-follow-multi/stop` с `{ groupId, fileId }`. Остановка всей группы: тот же эндпоинт без `fileId` или просто abort на стороне клиента (`AbortController`).

### SSH-пул соединений

Live и tail запускают exec-команды по SSH. Чтобы не упираться в лимиты sshd (`MaxStartups`, `MaxSessions`), сервер держит один SSH-клиент на сервер и мультиплексирует exec-каналы поверх него:

- `MAX_CHANNELS_PER_CLIENT = 8` — если каналов больше, открывается второй клиент.
- `IDLE_CLOSE_TIMEOUT_MS = 5000` — клиент закрывается через 5 с после освобождения последнего канала.
- При изменении кредов через `/api/config` старый пул разрушается.
- При `SIGTERM`/`SIGINT` все пулы закрываются корректно.

tz-selector.js ─► render.js ─────────────────────────────────────────┘
```

- **`utils.js`** — без зависимостей и без DOM. Тестируется в Node напрямую.
- **`virtual-list.js`** — монтирует только видимое окно строк (+ буфер), измеряет высоты, сохраняет раскрытое состояние `<details>`. Вызывается из `render.js`.
- **`state.js`** — экспортирует один объект `state` (мутации видны во всех модулях) и кэш DOM-ссылок `dom`.
- **`toast.js`** — самодостаточный, может импортироваться откуда угодно; контейнер `#toastContainer` создаётся на лету, если отсутствует.
- **`error-alerts.js`** — подключается через `handleNewLiveEntries`, вызываемый из `sse-client.js` после каждого `addLinesToLogs`. Хранит настройки в `localStorage`.
- **`sparkline.js`** — слушает `state.allLogs`, перерисовка дросселируется через `requestAnimationFrame`.
- **`tz-selector.js`** — экспортирует `getTzOffsetMinutes()`; при смене TZ вызывает `render()` для пересчёта колонки времени.
- Каждый «слой» вверх по стрелке имеет право вызывать всё, что под ним. Циклов нет.

### Зависимости (`package.json`)

- `express` — HTTP-сервер.
- `cors` — CORS-middleware.
- `ssh2-sftp-client` — SFTP-клиент (режим «Диапазон»).
- `ssh2` — низкоуровневый SSH-клиент (exec-команды `tail`, `grep`, `ls`).

Фронт **без сборщика и без runtime-зависимостей** — только ES-модули через `<script type="module">`.

```

### Что покрыто

**Unit (`test/utils.test.js`)** — чистые функции из `public/utils.js`:

- `traceIdColor`, `shortTraceId` — детерминированность, валидный диапазон hue, корректное обрезание длинных ID.
- `serviceColor` — детерминированность, валидный диапазон hue, отличие параметров HSL от `traceIdColor`, граничные значения (`null`/ `undefined`/`''`).
- `applyFilters` — поиск, уровни, временной диапазон, видимость сервисов, `traceFilter`, комбинирование, иммутабельность.
- `sortLogs` — все пять режимов (`time-asc`, `time-desc`, `service`, `level`, `trace`), контигуальность групп в режиме `trace`, поведение для записей без `traceId`, иммутабельность.
- `formatBytes`, `escapeHtml` — граничные случаи (0, null, undefined, спецсимволы HTML).
- `formatTimeFull`, `formatRelativeTime` — формат подсказки (4 строки, ISO/таймзона/миллисекунды), русское склонение one/few/many, граничные значения.
- `highlightMatch` — пустой query, регистронезависимое и непересекающееся выделение, экранирование HTML и литералов regex.
- `highlightJson` — раскраска ключей, строк, чисел, булевых, `null`; поиск поверх раскраски; не-JSON ввод; экранированные кавычки внутри ключа; границы слова для литералов.

## Развёртывание в production

### systemd-юнит

В корне лежит `logsviewer.service` — пример systemd-юнита:

Перед установкой сервиса перенесите на production-сервер всю папку проекта целиком. Зависимости (node_modules) лучше установить заранее:

```bash
npm install
```

В `logsviewer.service` укажите путь к этой папке в `WorkingDirectory`. Например, если проект лежит в `/var/logs-viewer`, используйте:

```
[Unit]
Description=Logs Viewer Service

[Service]
Type=simple
WorkingDirectory=/var/logs-viewer
ExecStart=/usr/bin/npm run start
Restart=on-failure
RestartSec=5s
KillSignal=SIGTERM
TimeoutStopSec=20

[Install]
WantedBy=multi-user.target
```

Установка:

```bash
sudo cp logsviewer.service /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl enable --now logsviewer
sudo systemctl status logsviewer
```

`SIGTERM` корректно завершает SSH-пулы (см. `gracefulShutdown` в `server.js`).

### Доступ к чужим логам

Скрипт `set_log_permissions.sh` помогает выдать пользователю `user` права на чтение логов в `/var/www/services/*-service/*.log` через ACL (`setfacl`) или, если ACL недоступно, через `chmod o+r`:

```bash
sudo ./set_log_permissions.sh
```

Параметры (`BASE_DIR`, `SERVICE_PATTERN`, `TARGET_USER`) задаются в начале скрипта.
