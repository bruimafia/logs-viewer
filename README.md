# Просмотрщик логов

Веб-приложение для просмотра и анализа лог-файлов в формате JSON Lines с поддержкой удалённых серверов по SSH/SFTP. Используется для логов микросервисов Перекоп-И 2.

## Основные возможности

- Просмотр локальных лог-файлов
- Просмотр логов с удалённых серверов по SSH/SFTP
- **Три режима загрузки удалённых файлов:**
  - **Хвост** — быстрая загрузка последних N строк с пагинацией
  - **Диапазон дат** — загрузка файла целиком с фильтрацией по периоду
  - **Live** — стриминг новых строк в реальном времени (`tail -F`)
- Объединение логов из нескольких файлов в общий хронологический поток
- Фильтрация по уровню (ERROR, WARN, INFO, DEBUG)
- Фильтрация по сервисам с **цветовой подсветкой и иконками-маркерами**
  у каждого сервиса (клик по чипу скрывает/показывает сервис)
- **Группировка и трассировка по `traceId` / `requestId`** — клик по бейджу
  трассы в любой записи фильтрует список до одной трассы; режим сортировки
  «По трассе» собирает записи одного `traceId` подряд
- Поиск по содержимому записей с подсветкой совпадений
- Фильтрация по временному диапазону (на стороне клиента)
- Сортировка по времени, сервису, уровню или трассе
- Множественные одновременные live-потоки с индивидуальной остановкой
- Переключение между тёмной и светлой темой (выбор сохраняется в браузере)
- **Toast-уведомления** об ошибках, успехах и подтверждениях вместо
  блокирующих `alert()` / `confirm()` — отображаются в правом нижнем
  углу, окрашены по уровню (error/warn/info/success), скрываются
  автоматически
- **Уведомления о новых ERROR** в live-режиме — браузерные desktop-уведомления
  (Notification API) и/или короткий звуковой сигнал (Web Audio API). Включаются
  галочками 🔔 / 🔊 в шапке, настройки сохраняются между сессиями.

## Требования

- **Node.js** 18 или выше (для production достаточно 16+; 18+ нужен только для `npm test` — встроенного `node:test`).
- **Целевые серверы:** Linux/Unix с утилитой `tail` (поддержка флагов `-n` и `-F`). Стандартный coreutils подходит. На Windows-серверах режимы «Хвост» и «Live» работать не будут.
- **SSH-доступ** на целевые серверы должен разрешать выполнение exec-команд (не только SFTP).
- **Современный браузер:** требуется поддержка ES-модулей (`<script type="module">`) — Chrome/Edge/Firefox/Safari последних 5 лет. Сборщик не используется.

## Установка

```bash
npm install
```

```bash
npm start
```

После запуска откройте браузер: <http://localhost:5000>

## Настройка удалённых серверов

Конфигурация удалённых серверов хранится в файле `remote-config.json` в корне проекта:

```json
{
  "servers": [
    {
      "id": "server1",
      "name": "Production Server",
      "host": "192.168.185.8",
      "port": 22,
      "username": "user",
      "password": "password",
      "files": [
        {
          "id": "file1",
          "name": "Order Service Log",
          "remotePath": "/var/services/order-service/order-service.log"
        },
        {
          "id": "file2",
          "name": "Person Service Log",
          "remotePath": "/var/www/services/person-service/person-service.log"
        }
      ]
    }
  ]
}
```

### Поля конфигурации сервера

| Поле       | Описание                                          |
| ---------- | ------------------------------------------------- |
| `id`       | Уникальный идентификатор сервера                  |
| `name`     | Отображаемое имя сервера                          |
| `host`     | IP-адрес или hostname сервера                     |
| `port`     | SSH-порт (обычно 22)                              |
| `username` | Имя пользователя для SSH                          |
| `password` | Пароль для SSH                                    |
| `files`    | Массив описаний доступных файлов                  |

### Поля описания файла

| Поле         | Описание                                        |
| ------------ | ----------------------------------------------- |
| `id`         | Уникальный идентификатор файла                  |
| `name`       | Отображаемое имя файла                          |
| `remotePath` | Полный абсолютный путь к файлу на сервере       |

При запросе конфигурации с фронта пароли заменяются на `••••••••`. При сохранении (если редактируете через API) маскированный пароль интерпретируется как «оставить прежний».

## Использование

### Локальные файлы

1. Нажмите **«Открыть файл(ы)»** и выберите один или несколько `.log` или `.json` файлов.
2. Если установлен флажок **«Добавить к открытым»**, новые файлы будут добавлены к уже загруженным; иначе — заменят текущий набор.

### Удалённые файлы

1. Нажмите кнопку **«Удаленные файлы»** — откроется модальное окно.
2. Сверху выберите **режим загрузки** (см. ниже).
3. Дождитесь автоматической проверки соединения с серверами (зелёный индикатор «Подключено»).
4. Отметьте нужные файлы галочками.
5. Нажмите кнопку загрузки внизу справа (текст меняется в зависимости от режима).

### Режим «Хвост» (по умолчанию)

Быстрая загрузка последних N строк через `tail -n N` по SSH. Не зависит от размера файла — даже из гигабайтного лога хвост загрузится за секунды.

- Параметр **«Строк»** (по умолчанию 1000) — сколько строк взять от конца файла.
- После загрузки в верхней части списка логов появляется кнопка **«↑ Загрузить ещё (более старые) ↑»**, которая подгружает следующую страницу той же длины для **всех** ранее загруженных в этом режиме файлов. Используется команда `tail -n (N+offset) | head -n N`.
- Если новых строк не получено — значит, достигнуто начало файла.

### Режим «Диапазон дат»

Полная загрузка файла через SFTP с фильтрацией по полю `time` каждой строки на стороне сервера. Подходит для исторических периодов, когда нужны все события за конкретный интервал.

- Поля **«От»** и **«До»** — границы периода (можно оставить пустыми).
- Если оба поля пустые — будет загружен весь файл.

### Режим «Live»

Стриминг новых строк в реальном времени через `tail -n N -F`. Флаг `-F` переживает ротацию логов: при пересоздании файла tail автоматически переоткроет его.

- Параметр **«Начальные строки»** (по умолчанию 100) — сколько последних строк показать в момент запуска. Можно поставить 0, чтобы видеть только новые события.
- После запуска в шапке появляется красный пульсирующий индикатор **`● LIVE [N] [Пауза] [Стоп]`**, где `N` — количество активных потоков.
- Клик по индикатору открывает список активных потоков с возможностью остановить каждый по отдельности.
- Кнопка **«Стоп»** на индикаторе останавливает все потоки сразу (с подтверждением).
- Кнопка **«Пауза»** временно останавливает обновление списка, **не разрывая SSE-соединение**: серверные процессы `tail -F` продолжают работать, новые строки копятся на стороне клиента в памяти. На кнопке появляется счётчик накопленных строк (`▶ Возобновить (+42)`), фон индикатора и точка приглушаются — визуальный сигнал, что отображение приостановлено. Клик по «Возобновить» применяет буфер: строки попадают в общий список и встают на свои места по времени. Удобно, чтобы спокойно прочитать или скопировать запись, не теряя поток. Если все потоки сами завершатся во время паузы — буфер автоматически применяется и пауза снимается. Кнопки «Стоп»/«Стоп всех» и «Очистить все» **отбрасывают** накопленный буфер без применения.
- **Оповещения об ERROR** (галочки 🔔 «Уведомления» и 🔊 «Звук» в шапке) — отдельно
  включают браузерные desktop-уведомления и короткий звуковой сигнал. Срабатывают
  только на записи уровня `ERROR` (а также `FATAL`/`CRITICAL`), и только пока live
  активен и **не на паузе**. Первый исторический батч строк (`initialLines`)
  алерт не запускает. Уведомления показываются только когда вкладка не на
  переднем плане (в фокусе — записи и так видно в списке). Чтобы не спамить
  при шквале ошибок, алерты дросселируются: уведомления — не чаще раза в 5
  секунд, звук — не чаще раза в 2 секунды. Настройки запоминаются в
  `localStorage` и переживают перезагрузку страницы.
- Модальное окно можно закрывать — потоки продолжают работать в фоне.
- Можно одновременно стримить **несколько файлов** с разных серверов.
- Новые записи кратко подсвечиваются зелёной вспышкой при появлении.
- При активных live-потоках общий буфер логов ограничен **50 000 записями** — самые старые отбрасываются автоматически (скользящее окно).
- Авто-прокрутка к новым записям работает, **пока вы не прокрутили список вручную**. После ручной прокрутки авто-скролл отключается; чтобы включить обратно — прокрутите к краю с новейшими записями.

### Совместное использование режимов

- **«Добавить к открытым»** работает во всех трёх режимах. Можно, например, загрузить хвост за вчерашний день и поверх него запустить live-стриминг для тех же или других файлов.
- При снятой галочке «Добавить к открытым» новая загрузка очищает всё, **в том числе останавливает все активные live-потоки** и **снимает активный фильтр по трассе** (он, скорее всего, относится к старому набору данных).
- Кнопка **«Очистить все»** в шапке также останавливает live-потоки, снимает фильтр по трассе и сбрасывает список.

### Фильтрация и сортировка

Панель фильтров под шапкой работает на стороне клиента поверх всех загруженных записей:

- **Поиск** — подстрочный поиск по полю `msg` и по сериализованной записи целиком (включая дополнительные поля). Поиск регистронезависимый; найденные подстроки подсвечиваются жёлтым фоном в столбце сообщения и в раскрывающемся блоке «Доп. поля».
- **Уровни** — чекбоксы ERROR / WARN / INFO / DEBUG.
- **С / До** — диапазон времени (формат `datetime-local` с точностью до секунды).
- **Быстрые диапазоны** — ряд кнопок-пресетов справа от полей «С / До»:
  **5 мин**, **15 мин**, **1 ч**, **6 ч**, **24 ч**, **7 дн**,
  **Сегодня** (от локальной полуночи до текущего момента), **Вчера**
  (полные сутки от полуночи до полуночи). Клик заполняет оба поля
  относительно текущего момента и подсвечивает выбранный пресет.
  Кнопка **×** очищает диапазон. Ручное редактирование полей «С / До»
  снимает подсветку — это значит, что отображаемый диапазон уже не
  совпадает с пресетом.
- **Сортировка** — по времени (ст→нов или нов→ст), по сервису, по уровню,
  по трассе (см. ниже).
- **Сервисы** — чипы сервисов: клик по чипу скрывает или показывает все
  логи этого сервиса. Зачёркнутый чип = скрытый сервис. У каждого сервиса
  свой **стабильный цвет и иконка** — и в чипе, и в столбце сервиса каждой
  строки лога (см. раздел «Подсветка цветом и иконками для сервисов»).

### Группировка и трассировка по `traceId` / `requestId`

Если запись содержит идентификатор трассы (см. распознаваемые поля ниже),
рядом с сообщением появляется небольшой **цветной бейдж** с укороченным
ID. Цвет стабильно генерируется из самого ID, поэтому записи одной
трассы визуально одного цвета.

- **Клик по бейджу** — включает фильтр «показать только эту трассу».
  Между панелью фильтров и списком появляется баннер
  `Фильтр по трассе: <id> [×]`. Кнопка **×** или повторный клик по тому же
  бейджу снимают фильтр.
- **Сортировка «По трассе»** — записи одного `traceId` собираются подряд
  и внутри группы упорядочиваются по времени. Между группами трассы
  идут по времени самой ранней записи каждой. Записи без `traceId`
  встраиваются как одиночные «группы» по своему времени — то есть не
  слипаются в одну смешанную пачку.

В каких полях ищется идентификатор трассы (в порядке приоритета —
побеждает первое непустое значение):

```
traceId  →  trace_id  →  requestId  →  request_id  →  correlationId  →  correlation_id
```

Поддерживаются строковые и числовые значения (числа приводятся к строке).
Если ни одного из этих полей в записи нет, бейдж не отображается,
а запись участвует в сортировке «По трассе» как одиночная.

Полный исходный `traceId` всегда виден в раскрывающемся блоке **«Доп. поля»**
— оттуда его удобно скопировать (например, для запроса в Jaeger/Tempo).

### Подсветка цветом и иконками для сервисов

Каждому сервису автоматически назначаются **стабильный цвет** и
**иконка-маркер** — оба значения детерминированно вычисляются из имени
сервиса (хэш-функция), поэтому при перезагрузке страницы и в разных
сессиях один и тот же сервис всегда выглядит одинаково.

Подсветка применяется одновременно в двух местах:

- **Чипы сервисов** в панели фильтров — цветной тинт фона, цветная
  граница, иконка перед именем.
- **Столбец «сервис»** в каждой строке лога — те же цвет и иконка,
  что в чипе фильтра. Это позволяет «связать глазами» строку и её
  сервис, не читая текст.

Палитра иконок состоит из **20 геометрических юникод-глифов** —
**10 базовых форм × 2 варианта заливки** (заполненная/контурная):

```
●○  ■□  ◆◇  ▲△  ▼▽  ◀◁  ▶▷  ★☆  ⬢⬡  ⬟⬠
```

Удвоение через варианты заливки даёт хорошее разнообразие: на 5 сервисах
вероятность одинаковых иконок ~28% (vs ~94% с прежней палитрой из 10), на
20 сервисах ожидается ~13 разных глифов из 20. Все глифы — из Unicode-блоков
Geometric Shapes (U+25xx) и Geometric Shapes Extended (U+2Bxx),
поддерживаются с Unicode 5.1 (2008).

Цвет — `hsl(hue, 55%, 50%)` с равномерно распределённым по 360° hue; параметры специально отличаются
от бейджей трасс (`hsl(hue, 60%, 55%)`), чтобы визуально сервисы и трассы
не путались.

Скрытый чип сервиса (после клика на «зачеркнуть») теряет цветовой
акцент и становится нейтрально-серым — иначе он продолжал бы привлекать
внимание, хотя его данных в списке не видно.

Для устаревших браузеров без поддержки `color-mix()` (старше ~2023 г.)
работает фолбэк: чипы и столбец сервиса отрисовываются стандартным
нейтральным цветом, иконки остаются на месте.

### Переключение темы

В правой части шапки расположена кнопка с иконкой солнца/луны, которая переключает интерфейс между **тёмной** (по умолчанию) и **светлой** темой. Выбор сохраняется в `localStorage` и применяется автоматически при следующем открытии страницы — до отрисовки контента, без «вспышки» неправильной темы.

### Уведомления

Все нефатальные ошибки и подтверждения показываются как **toast-уведомления**
в правом нижнем углу — поверх контента, но не блокируют интерфейс.

- **Ошибки** (красный) — например, неудачная загрузка хвоста, разрыв
  live-потока, ошибка SSH-соединения.
- **Предупреждения** (жёлтый) — для подтверждений деструктивных
  действий вроде «Остановить все live-потоки».
- **Инфо** (синий) — например, «Конец файла» в режиме «Загрузить ещё».
- **Успех** (зелёный) — в текущей реализации не используется по
  умолчанию, но доступен из API (`toast.success(…)`).

Тосты автоматически скрываются через несколько секунд (длительность
зависит от уровня); при наведении курсора таймер ставится на паузу.
Кнопка «×» закрывает уведомление вручную. Тосты-подтверждения с
кнопками «Подтвердить» / «Отмена» ждут явного ответа — `Esc` или
крестик равны «Отмена».

### Оповещения о новых ERROR в live-режиме

В дополнение к toast-уведомлениям, в live-режиме можно включить **активные
оповещения** о появлении записей уровня `ERROR` — даже если вкладка не на
переднем плане. Управляются двумя независимыми галочками в шапке:

- 🔔 **Уведомления** — браузерные desktop-уведомления через
  [Notification API](https://developer.mozilla.org/docs/Web/API/Notification).
  При первом включении браузер запросит разрешение. Показываются **только
  когда вкладка скрыта** (если вкладка активна — пользователь и так видит
  список). Заголовок: количество новых ERROR; тело: первая ошибка с именем
  сервиса. Клик по уведомлению фокусирует вкладку.
- 🔊 **Звук** — короткий двухтональный «бип» (880Hz → 660Hz, ~200 мс) через
  [Web Audio API](https://developer.mozilla.org/docs/Web/API/Web_Audio_API).
  Не использует внешние аудиофайлы. При включении сразу проигрывается
  тестовый сигнал, чтобы пользователь убедился, что звук слышен.

Оба оповещения:

- срабатывают **только при активном live-режиме**, без паузы;
- **игнорируют исторический хвост** при подключении (первая партия строк, она
  же `initialLines` — это уже произошедшие события);
- дросселируются, чтобы шквал ошибок не превратился в гудок и десятки
  всплывающих окон: уведомления — не чаще раза в 5 секунд, звук — не чаще
  раза в 2 секунды;
- запоминаются в `localStorage` и автоматически восстанавливаются после
  перезагрузки страницы.

Если уведомления заблокированы в настройках сайта на уровне браузера, галочка
автоматически блокируется с поясняющей подсказкой. Включить обратно можно
только вручную через настройки сайта в браузере (Chrome: 🔒 в адресной строке
→ «Уведомления»).

### Подсказка с полной временной меткой

Колонка с временем в каждой строке лога показывает компактное локализованное
время (формат `dd.mm.yyyy, hh:mm:ss`). При наведении курсора появляется
**нативная подсказка** с расширенной формой метки — это удобно, когда нужно
быстро увидеть миллисекунды, таймзону или относительное время без раскрытия
блока «Доп. поля».

Подсказка состоит из четырёх строк:

```
понедельник, 15 января 2024 г.
10:30:45.123 (UTC+03:00)
ISO: 2024-01-15T07:30:45.123Z
5 минут назад
```

- **День недели и полная дата** — на случай, когда время лежит на границе
  суток и непонятно, «какой это день».
- **Локальное время с миллисекундами** + явное смещение часового пояса —
  миллисекунды важны для быстрых event'ов, а UTC-смещение помогает, когда
  логи приехали с серверов в другой таймзоне.
- **ISO 8601** — удобно скопировать в фильтр другой системы (Jaeger/Tempo,
  Grafana, SQL-запрос).
- **Относительное время** — «5 минут назад», «2 часа назад», «через 30 секунд»
  (для записей из будущего, например с серверов с расходящимися часами).
  Граница «только что» — ±5 секунд.

Визуальные намёки: курсор меняется на `help`, текст времени слегка подчёркнут
точечной линией. Записи без поля `time` показываются как `—` и подсказку не
получают.

## Формат логов

Приложение ожидает логи в формате **JSON Lines** — один JSON-объект на строку:

```json
{"time":"2024-01-15T10:30:00.000Z","level":"INFO","msg":"Application started","service":"app"}
{"time":"2024-01-15T10:30:01.000Z","level":"ERROR","msg":"Connection failed","service":"db","error":"timeout","traceId":"5a8e1f4b-9c3d-4f6a-b2e8-1d7f0a3c9b5e"}
```

### Распознаваемые поля

| Поле             | Назначение                                                                                       |
| ---------------- | ------------------------------------------------------------------------------------------------ |
| `time`           | Временная метка в формате ISO 8601 (используется для сортировки и фильтра)                       |
| `level`          | Уровень логирования: `ERROR`, `WARN`, `INFO`, `DEBUG`                                            |
| `msg`            | Основное сообщение                                                                               |
| `service`        | Имя сервиса (опционально; если отсутствует — берётся имя файла)                                  |
| Поля трассировки | `traceId`, `trace_id`, `requestId`, `request_id`, `correlationId`, `correlation_id` — для группировки и фильтра «по трассе»  |

Все остальные поля показываются в раскрывающемся блоке **«Доп. поля»** под записью.

Строки, которые не парсятся как валидный JSON, **молча игнорируются** (не приводят к ошибкам, но и не отображаются).

## API сервера

Сервер на Express (порт 5000). Полезно знать, если хочется писать скрипты или интеграции.

| Метод | Путь                              | Назначение                                                                                |
| ----- | --------------------------------- | ----------------------------------------------------------------------------------------- |
| GET   | `/api/config`                     | Получить конфигурацию (пароли скрыты как `••••••••`)                                     |
| POST  | `/api/config`                     | Сохранить конфигурацию (`••••••••` в пароле = оставить прежний)                          |
| GET   | `/api/reload-config`              | Перечитать `remote-config.json` с диска                                                  |
| POST  | `/api/test-connection-by-id`      | Проверить SSH-соединение по `serverId`. Тело: `{ serverId }`                             |
| POST  | `/api/test-connection`            | Проверить SSH-соединение напрямую. Тело: `{ host, port, username, password }`            |
| POST  | `/api/stream-file`                | SSE: загрузка файла целиком через SFTP с опциональной фильтрацией по дате                |
| POST  | `/api/tail-file`                  | SSE: загрузка хвоста через `tail -n` с поддержкой пагинации (offset)                     |
| POST  | `/api/tail-follow`                | SSE: live-стриминг одного файла через `tail -F`                                          |
| POST  | `/api/tail-follow-multi`          | SSE: мультиплексный live-стриминг нескольких файлов одного сервера через одно соединение |
| POST  | `/api/tail-follow-multi/stop`     | Остановка отдельного канала или всей группы мультиплексного стрима                       |

### Параметры SSE-эндпоинтов

**`/api/stream-file`** — тело: `{ serverId, fileId, dateFrom?, dateTo? }`. События: `start`, `progress`, `lines`, `complete`, `error`.

**`/api/tail-file`** — тело: `{ serverId, fileId, lines, offsetLines? }`. На сервере выполняется `tail -n (lines+offsetLines) <path> | head -n lines`. События: `start`, `lines`, `complete`, `error`.

**`/api/tail-follow`** — тело: `{ serverId, fileId, initialLines? }`. На сервере выполняется `tail -n initialLines -F <path>`. События: `start`, `lines`, `info`, `error`, `end`. Каждые 25 секунд шлётся SSE-комментарий `: keepalive` для поддержания соединения через прокси. При закрытии соединения клиентом серверу отправляется `SIGTERM` для процесса `tail`.

**`/api/tail-follow-multi`** — тело: `{ serverId, files: [{ fileId, initialLines? }, ...] }`. Открывает **одно** SSE-соединение, в котором мультиплексируются стримы нескольких файлов одного сервера. Простаивающие соединения закрываются через 5 секунд после освобождения. На уровне HTTP несколько live-потоков одного сервера передаются через одно SSE-соединение (`/api/tail-follow-multi`) — это обходит лимит браузера в 6 параллельных HTTP/1.1 соединений на origin. Один сервер ↔ один HTTP-стрим ↔ 1–2 SSH-коннекта, независимо от числа отслеживаемых файлов.

### Структура фронтенд-модулей

Фронт устроен как набор ES-модулей с однонаправленным графом импортов:

```
utils.js  ──────────►  render.js  ──►  sse-client.js  ──►  remote-modal.js  ──►  app.js
state.js  ──────────────┘            ──────┘            ──────┘                ─────┘
                                        ▲                   ▲                    ▲
                                        └─── toast.js ──────┴────────────────────┘
                                        └─── error-alerts.js ─────────────────────┘
```

- **`utils.js`** — без зависимостей и без DOM. Тестируется в Node напрямую.
- **`state.js`** — экспортирует один объект `state` (мутации видны во всех модулях) и кеш DOM-ссылок `dom`.
- **`toast.js`** — самодостаточный модуль toast-уведомлений
  (`toast.{success,info,warn,error}` и `toastConfirm`). Не зависит от
  других модулей фронта и может импортироваться откуда угодно;
  контейнер `#toastContainer` создаётся на лету, если отсутствует
  в разметке.
- **`error-alerts.js`** — браузерные уведомления (Notification API) и
  короткий звуковой сигнал (Web Audio API) на новые ERROR-записи в
  live-режиме. Подключается к потоку через `handleNewLiveEntries`,
  вызываемый из `sse-client.js` после каждого `addLinesToLogs`. Хранит
  настройки в `localStorage`, состояние дросселирования — в локальных
  переменных модуля.
- Каждый «слой» вверх по стрелке имеет право вызывать всё, что под ним. Циклов нет.

Зависимости (см. `package.json`):

- `express` — HTTP-сервер
- `cors` — CORS-middleware (на случай отдельного хоста для фронта)
- `ssh2-sftp-client` — SFTP-клиент (для режима «Диапазон дат»)
- `ssh2` — низкоуровневый SSH-клиент (для exec-команд `tail` и `tail -F`)

## Тестирование

Тесты используют встроенный test-runner Node ([`node:test`](https://nodejs.org/api/test.html)) — никаких внешних библиотек (jest/mocha/chai) не нужно. Требуется Node.js 18+.

```bash

npm run test:server # только HTTP-эндпоинты
```

### Что покрыто

**Unit (`test/utils.test.js`)** — чистые функции из `public/utils.js`:

- `parseLogLine` — корректный JSON, пустые/невалидные строки, выбор `_serviceKey`, обработка пропущенного/невалидного `time`, извлечение `_traceId` из `traceId`/`request_id`.
- `getTraceId` — приоритет полей, snake_case-варианты, числовые ID, пропуск пустых/`NaN`/массивов/объектов/булевых, кастомный список полей.
- `traceIdColor`, `shortTraceId` — детерминированность по traceId, валидный диапазон hue, корректное обрезание длинных ID.
- `serviceColor`, `serviceIcon`, `SERVICE_ICON_PALETTE` — детерминированность
  по имени сервиса, валидный диапазон hue, отличие параметров HSL от
  `traceIdColor`, граничные значения (`null`/`undefined`/`''`),
  структура палитры «10 форм × 2 заливки» (спот-чек заполненных и
  контурных глифов), уникальность элементов, разнообразие иконок
  на 40 разных именах (порог недостижимый со старой палитрой ≤10).
- `applyFilters` — поиск по `msg` и доп. полям, фильтр по уровням, временной диапазон, видимость сервисов, фильтр по `traceFilter`, комбинирование фильтров, иммутабельность.
- `sortLogs` — все пять режимов (`time-asc`, `time-desc`, `service`, `level`, `trace`), включая порядок неизвестных уровней, контигуальность групп в режиме `trace`, поведение для записей без `traceId`, иммутабельность.
- `formatBytes`, `escapeHtml` — граничные случаи (0, null, undefined, спецсимволы HTML).
- `formatTimeFull`, `formatRelativeTime` — формат полной подсказки (4 строки,
  ISO/таймзона/миллисекунды), русское склонение «секунда/минута/час/день» по
  правилам one/few/many, граничные значения (`null`, `0`, `NaN`).
- `highlightMatch` — пустой/null query, регистронезависимое и непересекающееся выделение, экранирование HTML и литералов regex.
- `getQuickRange`, `msToDatetimeLocalValue` — все пресеты (5m/15m/1h/6h/24h/7d/today/yesterday), fallback для неизвестного пресета, дефолт `Date.now()`, форматирование в локальное время, padding одноцифровых компонент, round-trip через `<input type="datetime-local">`.

**Integration (`test/server.test.js`)** — поднимает реальный `server.js` на свободном порту и проверяет:

- статика отдаётся (`GET /` → `index.html`, `/public/styles.css`, `/public/utils.js`),
- 404 на несуществующий путь,
- `/api/config` возвращает JSON и маскирует пароли как `••••••••`,
- `/api/reload-config` отвечает `success: true`,
- валидация входных параметров (`/api/test-connection-by-id` с несуществующим id, `/api/tail-follow-multi` без обязательных полей).
