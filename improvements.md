# Улучшения и новые функции для просмотрщика логов

Я изучил проект — это просмотрщик JSON-логов микросервисов «Перекоп-И 2» с поддержкой удалённых файлов через SSH/SFTP, live-стримингом через `tail -F`, мультиплексированным SSE и тремя режимами загрузки. Ниже подробный список идей, разбитый по категориям. Я постарался формулировать каждую идею так, чтобы её можно было сразу превратить в задачу: с кратким описанием сути, файлами, которых она коснётся, и (где уместно) указанием приоритета и оценки сложности.

---

## 1. Безопасность и аутентификация (высокий приоритет)

**1.1. Аутентификация веб-интерфейса.** Сейчас любой, кто имеет сетевой доступ к порту 5000, видит все логи продакшна. Добавить базовую аутентификацию (логин/пароль или OIDC/LDAP, если есть корпоративный SSO), хранить сессии в подписанных cookie, защитить все `/api/*` endpoint'ы middleware. Файлы: `server.js`, новый `auth.js`, добавить страницу логина в `index.html`. Сложность: средняя.

**1.2. Шифрование паролей в `remote-config.json`.** Сейчас пароли лежат в plain text — это явно отмечено в README как проблема. Минимум — шифрование AES-256 с мастер-ключом из переменной окружения `LOG_VIEWER_SECRET`. Лучше — поддержка системного keychain (на сервере) или интеграция с HashiCorp Vault. Сложность: средняя.

**1.3. Поддержка SSH-ключей.** Сейчас только парольная аутентификация. Добавить в конфиг поля `privateKeyPath` / `passphrase` и поддержку ED25519/RSA через `ssh2`. Опционально — загрузка ключей через UI с сохранением в защищённую папку. Сложность: средняя.

**1.4. HTTPS-сервер.** Добавить опцию запуска по HTTPS с указанием путей к сертификату/ключу через переменные окружения или конфиг. Без HTTPS любая аутентификация — иллюзия защиты. Сложность: низкая.

**1.5. Rate limiting на API.** Защита от перегрузки и DOS — особенно важно для SSE-эндпоинтов, которые держат долгие соединения. Использовать `express-rate-limit` с раздельными лимитами для тяжёлых endpoint'ов (`/api/stream-file`) и лёгких (`/api/config`). Сложность: низкая.

**1.6. Аудит-лог действий.** Записывать, кто и когда запрашивал какие файлы с каких серверов, в отдельный файл `audit.log`. Минимально — IP, user-agent, время, операция, serverId/fileId. Сложность: низкая.

## 2. Функциональные возможности работы с логами (высокий приоритет)

**2.1. Группировка и трассировка по `traceId` / `requestId`.** Это, на мой взгляд, самая ценная функция для отладки микросервисов. Добавить:
- автоматическое распознавание поля `traceId` (или настраиваемое имя поля) в JSON-записях;
- режим отображения «по трассе» — все записи одного `traceId` собраны вместе, отсортированы по времени, визуально связаны;
- клик по `traceId` в любой записи → фильтр «показать только эту трассу».

Файлы: `index.html` (новая логика рендера + фильтра). Сложность: средняя.

**2.2. Расширенный синтаксис поиска (KQL-подобный).** Сейчас поиск — простой substring. Добавить выражения типа `level:ERROR AND service:assistant AND msg:"timeout"`, с операторами `AND`/`OR`/`NOT`, диапазонами `time:[now-1h TO now]`, регулярками `msg:/regex/`. Реализуется на клиенте поверх `allLogs`. Файл: `index.html`. Сложность: средняя–высокая.

**2.3. Регулярные выражения в поиске (минимальный вариант 2.2).** Если полноценный KQL — это много работы, как первый шаг добавить чекбокс «regex» рядом с полем поиска и игнор-кейс-чекбокс. Сложность: низкая.

**2.4. Подсветка совпадений в результатах поиска.** Найденные подстроки в `msg` выделять жёлтым фоном. Помогает быстро ориентироваться, особенно при широких запросах. Сложность: низкая.

**2.5. Сохранение пресетов фильтров.** Кнопка «Сохранить как пресет» рядом с тулбаром: имя + текущая комбинация (поиск, уровни, сервисы, сортировка). Пресеты — в `localStorage`, выпадающий список «Применить пресет». Полезные дефолты: «Только ERROR», «ERROR+WARN за последний час», «Auth/Assistant сервисы». Сложность: низкая.

**2.6. Быстрые временные диапазоны.** Кнопки-чипы «Последние 5 мин / 15 мин / 1 ч / 24 ч / Сегодня» рядом с полями `С / До`. Сейчас приходится вручную вводить datetime — очень неудобно. Сложность: низкая.

**2.7. Поделиться видом через URL.** Сериализовать текущие фильтры/сортировку/выбранные файлы в query-параметры или hash. Открывая ссылку, коллега увидит ровно тот же набор. Файлам с локального диска ссылка, очевидно, не поможет — но для удалённых сработает. Сложность: средняя.

**2.8. Экспорт отфильтрованных логов.** Кнопка «Скачать» с выбором формата: JSON Lines (как есть), CSV (плоский, основные поля), Markdown (для отчётов об инцидентах). Сложность: низкая.

**2.9. Закладки на записи.** Клик по иконке-флажку слева от строки помечает её; кнопка «Только закладки» в тулбаре фильтрует. Закладки хранятся в `localStorage` по композитному ключу (`time + service + msg-hash`). Сложность: средняя.

**2.10. Контекст вокруг записи.** Кнопка «±20 строк контекста» в раскрытой записи — подгружает соседние записи того же файла/сервиса игнорируя текущий фильтр. Сложность: средняя.

**2.11. Diff-режим между двумя временными диапазонами.** Параллельная панель: «было в 14:00–14:05» vs «стало в 15:00–15:05» того же сервиса. Помогает находить регрессии. Сложность: высокая.

## 3. Live-режим и оповещения (средний приоритет)

**3.1. Пауза/возобновление live-потока без разрыва.** Сейчас можно только остановить. Добавить кнопку «Пауза» — клиент перестаёт добавлять входящие строки в `allLogs`, но соединение живо; кнопка «Возобновить» применяет накопленный буфер. Сложность: низкая.

**3.2. Браузерные уведомления о новых ERROR.** Опциональная настройка «Уведомлять при ERROR/WARN в live-режиме» с запросом `Notification.requestPermission()`. Полезно, когда страница в фоне. Сложность: низкая.

**3.3. Звуковой сигнал на ERROR.** Опционально, в паре с 3.2. Сложность: низкая.

**3.4. Мини-спарклайн со статистикой.** Узкий канвас под тулбаром, который показывает количество записей по уровням за последние N минут (стэкбар). Обновляется в live-режиме, помогает увидеть спайк. Сложность: средняя.

**3.5. Автореконнект live-потока.** При обрыве SSE-соединения (сеть, перезапуск прокси) — автоматически переподключаться с экспоненциальной задержкой, продолжая с момента последней полученной строки. Сложность: средняя.

**3.6. Multi-server live.** Сейчас `/api/tail-follow-multi` мультиплексирует только файлы одного сервера. Часто нужно следить за логами Stage и Production одновременно — открывается два SSE-соединения, это нормально. Но в UI индикатор live должен показывать оба сервера, и «Стоп всех» должен останавливать оба. Уже частично сделано? Проверить. Сложность: низкая.

## 4. Парсинг и поддержка форматов (средний приоритет)

**4.1. Многострочные записи (stack traces).** Сейчас строки, которые не парсятся как JSON, молча игнорируются. Если приложение пишет stack trace в поле `stack` или `error` внутри JSON — это уже отображается. Но если stack trace идёт отдельными строками после JSON-записи — они теряются. Добавить логику: непарсящиеся строки прикрепляются как продолжение к предыдущей записи. Сложность: средняя.

**4.2. Подсветка синтаксиса в раскрытых записях.** В блоке «Доп. поля» рендерить JSON с подсветкой (ключи, строки, числа, null) — например, через лёгкий self-made highlighter или `highlight.js`. Сложность: низкая.

**4.3. Дерево JSON для глубоко вложенных полей.** Вместо `<pre>` с сериализованным JSON — сворачиваемое дерево (как в Chrome DevTools). Особенно ценно для записей с большой структурой `payload` или `context`. Сложность: средняя.

**4.4. Поддержка не-JSON форматов.** Многие сервисы пишут logfmt (`key=value key2="value 2"`) или просто plain text. Добавить выбор парсера в модалке загрузки или авто-определение по первой строке. Сложность: средняя.

**4.5. Поддержка `.log.gz` / `.log.bz2`.** Архивные логи обычно сжаты. Уметь распаковывать на стороне сервера (zlib для gz) и передавать клиенту распакованным. Сложность: низкая.

**4.6. Конфигурируемые поля парсера.** В UI указать «поле времени = `ts`», «поле уровня = `severity`» — если конкретный сервис называет поля иначе, чем `time/level/msg/service`. Сложность: средняя.

## 5. Производительность и архитектура (средний приоритет)

**5.1. Виртуализация списка записей.** Сейчас все строки рендерятся в DOM. С `LIVE_BUFFER_CAP = 50000` это десятки тысяч узлов — на средней машине заметно тормозит скролл и `render()`. Внедрить виртуальный скролл (рендерить только видимые ±30 строк). Можно без библиотек: `IntersectionObserver` + сегменты по 100. Файл: `index.html`. Сложность: средняя–высокая, но критично.

**5.2. Серверная фильтрация для режима «диапазон дат».** Сейчас при выборе диапазона сервер тянет весь файл через SFTP, а фильтрация по дате — на клиенте? Стоит проверить (`/api/stream-file` принимает `dateFrom/dateTo`). Если фильтр уже на сервере — добавить ещё grep по уровню (`grep -E '"level":"(ERROR|WARN)"'`) чтобы не тащить миллионы строк INFO. Сложность: низкая.

**5.3. Серверный поиск (grep) перед загрузкой.** В модалке режима «Хвост» и «Диапазон» опциональное поле «Содержит» — выполняется на сервере как `grep`, до `tail`. Радикально ускоряет работу с большими файлами. Сложность: низкая.

**5.4. Кэш загруженных файлов на сервере.** Для одного и того же `serverId/fileId/dateFrom/dateTo` отдавать кэшированный результат с TTL ~30 сек. Полезно при перезагрузке страницы или одинаковых запросах разных пользователей. Сложность: средняя.

**5.5. Gzip-сжатие SSE-ответов.** Добавить `compression` middleware с проверкой совместимости — SSE через gzip нужно отдельно тестировать (buffering ломает). Сложность: средняя, нужна аккуратность.

**5.6. Модуляризация `index.html`.** Сейчас всё (CSS + JS на несколько тысяч строк + HTML) в одном файле. Разделить на `public/styles.css`, `public/app.js`, `public/sse-client.js`, `public/render.js`, `public/remote-modal.js`. Это сильно облегчит сопровождение. Сложность: средняя.

**5.7. Миграция на TypeScript.** В паре с 5.6. Особенно полезно для модели данных (`LogEntry`, `SseEvent`, `RemoteConfig`) и SSE-парсера, где сейчас ошибки ловятся только в рантайме. Сложность: высокая, но даёт большой выигрыш.

**5.8. Тесты.** В проекте, судя по `package.json`, тестов нет. Добавить хотя бы:
- unit-тесты на `parseLogLine`, фильтрацию, сортировку (jest или node:test);
- интеграционные тесты на API с mock SSH (ssh2 поддерживает локальный сервер для тестов);
- E2E с playwright для основных сценариев (открыть файл, переключить тему, применить фильтр).

Сложность: средняя, но окупается быстро.

## 6. UI и UX (средний приоритет)

**6.1. Замена `alert()` на toast-уведомления.** Сейчас ошибки показываются нативным `alert`, который блокирует UI и выглядит чужеродно. Простая система: правый нижний угол, 4-5 секунд, цвета по уровню (error/warning/success/info). Сложность: низкая.

**6.2. Боковая панель деталей вместо `<details>`.** Клик по записи → справа выезжает панель с полным JSON, метаданными (файл, сервер, размер записи), кнопками «Скопировать», «Закладка», «Показать трассу». Сейчас раскрытие происходит inline и ломает выравнивание. Сложность: средняя.

**6.3. Подсказки (tooltip) на полной временной метке.** Сейчас время показывается локализованным форматом с точностью до секунды. На hover показывать ISO 8601 с миллисекундами и часовым поясом. Сложность: низкая.

**6.4. Селектор часового пояса.** Выпадающий список «UTC / Локальное / +03:00 …» — особенно нужно, когда логи с серверов в одном TZ, а инженер в другом. Сложность: низкая.

**6.5. Подсветка цветом и иконками для сервисов.** Авто-назначение стабильного цвета на основе хэша имени сервиса (как в Jaeger). Опционально — переопределение в конфиге (`"color": "#ff8800"`). Сложность: низкая.

**6.6. Группировка сервисов в чипах.** В `remote-config.json` ввести поле `group` для файлов («infra», «business», «storage»). В тулбаре чипы группируются с заголовками и кнопками «Скрыть всю группу». Сложность: низкая.

**6.7. Компактный режим отображения.** Чекбокс «Компактно» — уменьшает паддинги, шрифт до 11 px, прячет столбец сервиса в иконку. Сложность: низкая.

**6.8. Wrap / no-wrap для длинных сообщений.** Сейчас длинные `msg` переносятся, ломая выравнивание. Кнопка-переключатель «Перенос строк» (как в IntelliJ). Сложность: низкая.

**6.9. Кастомизация колонок.** Чекбоксы «Показать колонку: Время / Уровень / Сервис / Сообщение / поле X». Дать вынести любое extra-поле как отдельную колонку. Сложность: средняя.

**6.10. Горячие клавиши.**
- `/` — фокус в поле поиска;
- `Esc` — очистить поиск / закрыть модалку;
- `j` / `k` — следующая/предыдущая запись (с подсветкой);
- `e` — раскрыть/свернуть «Доп. поля» выделенной записи;
- `t` — переключить тему;
- `g g` — в начало списка, `G` — в конец.

Файл: `index.html`. Сложность: средняя.

**6.11. Drag-and-drop локальных файлов в любое место страницы.** Сейчас нужно жать на кнопку. Добавить общий overlay при перетаскивании. Сложность: низкая.

**6.12. Запоминание состояния UI.** Сохранять в `localStorage` последние выбранные уровни, режим append, режим загрузки, числа строк, развёрнутые группы серверов. Сложность: низкая.

**6.13. Адаптивная вёрстка для мобильных/узких экранов.** Тулбар сейчас широкий и горизонтально сворачивается так себе. Сделать collapsible-секции «Фильтры», «Сервисы» с кнопкой-аккордеоном. Сложность: средняя.

**6.14. Доступность (a11y).** ARIA-атрибуты на чипы (`role="button"`, `aria-pressed`), фокусы видимы, чекбоксы и кнопки нативные где можно, контраст текста в обеих темах проверить через WCAG AA. Сложность: средняя.

**6.15. Onboarding/всплывающие подсказки** при первом запуске — короткий тур по тулбару и модалке. Можно через `intro.js` или собственный простой компонент. Сложность: низкая.

## 7. Управление конфигурацией (средний приоритет)

**7.1. UI-редактор `remote-config.json`.** Сейчас, насколько я вижу, серверы и файлы редактируются вручную в JSON. Сделать страницу настроек с формами: «Добавить сервер», «Добавить файл», inline-rename, drag-to-reorder. Уже есть `/api/config` POST — нужно использовать. Сложность: средняя.

**7.2. Glob-паттерны путей.** В поле `remotePath` поддержать `/var/log/services/*/*.log` — сервер на каждом обращении делает `ls` и подставляет реальные файлы. Полезно, когда сервисы добавляются часто. Сложность: средняя.

**7.3. Браузер удалённой файловой системы.** В модалке «Удалённые файлы» добавить вкладку «Обзор» — навигация по директории, выбор любого файла на лету (без добавления в конфиг). Сложность: средняя.

**7.4. Индикатор состояния серверов в шапке.** Кружок зелёный/жёлтый/красный рядом с именем каждого сервера, обновляется раз в минуту. Сейчас проверка только по явному клику в модалке. Сложность: низкая.

## 8. Интеграции и наблюдаемость (низкий приоритет, но «нашпигованно» полезно)

**8.1. Структурированный анализ ошибок.** Парсить поле `error`/`stack` отдельно: распознавать класс исключения и формировать сводку «X встретилось N раз». Бонус — кнопка «Сгруппировать похожие ошибки». Сложность: высокая.

**8.2. Webhook на паттерн.** В настройках задать паттерны (regex или KQL) и URL — при совпадении в live-режиме отправлять POST. Помогает интегрироваться с Mattermost/Slack/PagerDuty. Сложность: средняя.

**8.3. Ссылки в Grafana/Kibana/Jaeger.** В записях с `traceId` — кнопка «Открыть в Jaeger» (URL-шаблон из настроек). Аналогично для дашбордов по `service`. Сложность: низкая.

**8.4. PWA + offline.** Service worker, манифест, кэш загруженных логов. Можно открыть сессию без сети для анализа. Сложность: средняя.

**8.5. Docker-образ.** Готовый `Dockerfile` и `docker-compose.yml` с volume для `remote-config.json` и пробросом порта. Сейчас, видимо, разворачивается вручную. Сложность: низкая.

## Что я бы делал первым

Если упорядочить по соотношению «польза / трудозатраты», я бы предложил такой топ-7 для ближайших спринтов:

1. **Виртуализация списка** (5.1) — без этого UI деградирует на больших объёмах, а с live-потоками это случится быстро.
2. **Группировка по `traceId`** (2.1) — главное конкурентное преимущество просмотрщика логов микросервисов.
3. **Toast-уведомления вместо `alert`** (6.1) и **быстрые временные диапазоны** (2.6) — самые заметные UX-улучшения за минимальные усилия.
