# Анализ и рекомендации по улучшению Logs Viewer

**Дата:** май 2026  
**Версия проекта:** 1.0.0  
**Статус:** Production-ready приложение для просмотра JSON Lines логов

---

## Часть 1: Анализ текущего состояния

### 1.1 Сильные стороны проекта

#### ✅ Архитектура и дизайн
- **Простота развёртывания**: Single Node.js процесс, нет build-пайплайна, работает из коробки
- **Минимальные зависимости**: Только `express`, `cors`, `ssh2`, `ssh2-sftp-client`
- **Модульная структура фронтенда**: Чистое разделение concerns (state, utils, render, SSE)
- **No-bundler подход**: ES-модули напрямую в браузер — меньше сложности
- **Грамотная обработка ошибок**: Корректное завершение при SIGTERM, закрытие SSH-пулов

#### ✅ Функциональность
- **Три режима загрузки**: tail (pagination), date-range (streaming), live (SSE)
- **SSH-пул соединений**: До 8 каналов на соединение, auto-cleanup через 5с
- **Live-трассировка**: Группировка по traceId/requestId/correlationId с стабильной раскраской
- **Спарклайн активности**: Интерактивная гистограмма по уровням с клинами для фильтра
- **Доступность**: Skip-link, ARIA-роли, pointers-reduced-motion, live-regions
- **Мультиязычность**: Весь UI на русском с правильным склонением

#### ✅ UX
- **Комбинированные фильтры**: Поиск + уровни + временной диапазон + сервисы
- **Тёмная/светлая тема**: Сохраняется в localStorage, применяется до первого paint
- **Toast-уведомления**: Неблокирующие, с разными уровнями severity
- **Desktop alerts**: Notification API + Web Audio (звуковой сигнал на ERROR)
- **Релативное время**: «5 минут назад» с русским склонением

#### ✅ DevOps и deployment
- **Встроенные тесты**: `node:test` (встроенный в Node 18+)
- **systemd unit**: Готовый конфиг для Linux production
- **Управление правами**: `set_log_permissions.sh` для доступа к чужим логам
- **Graceful shutdown**: Корректное закрытие SSH-соединений на SIGTERM

### 1.2 Слабые стороны и боль-пойнты

#### ⚠️ Безопасность
- **Пароли в конфигурации**: `remote-config.json` хранит SSH-пароли в открытом виде
- **Нет аутентификации**: Любой, кто может открыть localhost:5000, получает доступ ко всем логам
- **CORS включён везде**: `app.use(cors())` без ограничений — уязвимость для cross-site-attacks
- **Shell injection**: Хотя используется `shellEscape`, риск при обработке глобов или путей
- **Нет rate-limiting**: Возможна DOS через много параллельных SSH-запросов

#### ⚠️ Масштабируемость
- **Буфер в памяти**: 50 000 записей в live-режиме — может быть недостаточно/переполнить память
- **Одноточечный отказ**: Нет кластеризации, балансировки; сервер не масштабируется горизонтально
- **SSH-соединения**: При наличии 100 серверов может быть проблема с файл-дескрипторами
- **Нет кэширования**: Каждый запрос делает новое SFTP-соединение

#### ⚠️ Performance
- **Синхронная обработка JSON**: Парсинг больших файлов на фронте может зависнуть
- **Нет virtual scrolling**: При миллионах записей список станет неответственным
- **grep on every tail**: Серверный grep выполняется за каждый pagination запрос
- **Live-режим без оптимизаций**: Отправка всех полей каждой записи, нет сжатия

#### ⚠️ Наблюдаемость (Observability)
- **Нет логирования**: Ошибки SSH видны только в `console.error`, нет структурированного логирования
- **Нет метрик**: Нет информации о количестве подключений, времени отклика, объёме данных
- **Нет трейсинга**: Невозможно отследить, какой запрос к какому серверу идёт
- **Нет health-check**: Нет API для проверки жизнеспособности приложения

#### ⚠️ Тестирование
- **Недостаточное покрытие**: Нет e2e-тестов, нет mock для SSH/SFTP
- **Smoke-тесты** только базовые: не проверяют streaming, SSE, мультиплексирование
- **Нет negative-тестов**: Нет проверки поведения при разрывах соединения, timeout'ах, невалидных данных

#### ⚠️ Мониторинг и отладка
- **Нет визуализации логов в памяти**: Как понять, что происходит в live-буфере?
- **Нет встроенной аналитики**: Какие файлы открывают чаще? Какие серверы недоступны?
- **Нет версионирования конфига**: История изменений remote-config.json не сохраняется

## Часть 2: Рекомендации по улучшению

### 2.1 Безопасность (HIGH PRIORITY)

#### 📌 2.1.1 Базовая аутентификация
**Проблема**: Любой может открыть приложение и увидеть все логи.  
**Решение**:
- Добавить простую аутентификацию (Basic Auth или JWT)
- Конфиг с хэшем пароля: `{ password: "sha256:abc123..." }`
- API `/api/login` для получения токена
- Middleware для проверки токена на всех protect-маршрутах

```javascript
// Пример: Basic Auth через middleware
app.use((req, res, next) => {
  const auth = req.headers.authorization?.split(' ')[1];
  if (!auth) return res.status(401).json({ error: 'Unauthorized' });
  const [user, pass] = Buffer.from(auth, 'base64').toString().split(':');
  if (pass === CONFIG.password) next();
  else res.status(401).json({ error: 'Invalid credentials' });
});
```

**Приоритет**: 🔴 Критичный

#### 📌 2.1.2 Безопасное хранение паролей SSH
**Проблема**: Пароли хранятся в открытом виде в JSON.  
**Решение**:
- Использовать `process.env.SSH_PASSWORD_<SERVER_ID>` вместо файла
- Или шифровать пароли в `remote-config.json` с ключом из ENV
- Или использовать SSH-ключи вместо паролей (рекомендуется)

```json
{
  "servers": [{
    "id": "prod",
    "host": "prod.example.com",
    "username": "logs-user",
    "auth": "key",  // или "password"
    "keyPath": "/home/user/.ssh/id_rsa",  // для SSH-ключей
    "passwordEnvVar": "SSH_PASSWORD_PROD"  // для паролей
  }]
}
```

**Приоритет**: 🔴 Критичный

#### 📌 2.1.3 Ограничение CORS
**Проблема**: `cors()` разрешает запросы с любых источников.  
**Решение**:
```javascript
app.use(cors({
  origin: process.env.ALLOWED_ORIGINS?.split(',') || ['http://localhost:5000'],
  credentials: true,
  methods: ['GET', 'POST'],
  allowedHeaders: ['Content-Type', 'Authorization']
}));
```

**Приоритет**: 🟡 Высокий

#### 📌 2.1.4 Rate-limiting
**Проблема**: Нет защиты от DOS через множество SSH-запросов.  
**Решение**:
```bash
npm install express-rate-limit
```
```javascript
const rateLimit = require('express-rate-limit');
const limiter = rateLimit({
  windowMs: 1 * 60 * 1000, // 1 минута
  max: 100, // 100 запросов
  keyGenerator: (req) => req.headers.authorization || req.ip
});
app.use('/api/', limiter);
```

**Приоритет**: 🟡 Высокий

#### 📌 2.1.5 Валидация входных данных
**Проблема**: Мало проверки типов и диапазонов в API.  
**Решение**:
```bash

```

#### 📌 2.3.2 Кэширование результатов SSH-команд
**Проблема**: Каждый запрос за одинаковые данные создаёт новый SSH-запрос.  
**Решение**:
```javascript
const NodeCache = require('node-cache');
const cache = new NodeCache({ stdTTL: 300 }); // 5 минут

async function getFileListWithCache(serverId, pattern) {
  const cacheKey = `glob:${serverId}:${pattern}`;
  const cached = cache.get(cacheKey);
  if (cached) return cached;
  
  const result = await expandGlobOnServer(serverId, pattern);
  cache.set(cacheKey, result);
  return result;
}
```

**Приоритет**: 🟡 Высокий

#### 📌 2.3.3 Сжатие данных в SSE
**Проблема**: Передаём полный JSON каждой записи, много трафика.  
**Решение**:
```javascript
// Отправлять только изменённые поля + ID
const LINE_FORMATS = {
  minimal: ['id', 'time', 'level', 'msg', 'service', '_traceId'],
  full: 'all'
};

function serializeLogLine(log, format = 'minimal') {
  if (format === 'minimal') {
    return {
      id: log._id,
      t: log.time,  // t = time
      l: log.level, // l = level
      m: log.msg,
      s: log.service,
      tr: log._traceId
      // Остальное передать отдельно при expand
    };
  }
  return log;
}
```

**Приоритет**: 🟢 Средний (если трафик большой)

#### 📌 2.4.2 Worker threads для парсинга больших файлов
**Проблема**: Парсинг JSON блокирует event loop.  
**Решение**:
```javascript
const { Worker } = require('worker_threads');

function parseLogLinesInWorker(lines) {
  return new Promise((resolve, reject) => {
    const worker = new Worker('./parse-worker.js');
    worker.on('message', resolve);
    worker.on('error', reject);
    worker.postMessage({ lines });
  });
}
```
